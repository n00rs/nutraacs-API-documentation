NutraACS Integration Guide – Save Third Party Multi-Voucher Invoice API

Push a mixed batch of vouchers (any combination of Car, Ticket, Hotel, and Other) into NutraACS in a single request, and receive one consolidated invoice covering all of them.

Use this endpoint when a single trip or booking spans multiple product types (e.g. flight + hotel + car for the same customer) and you want them settled as one invoice. For single-type batches you may keep using the per-type endpoints (Ticket, Car, Hotel, Other).

1. API Endpoint

POST /api/integration/voucher/save_third_party_multi_voucher_invoice

Content-Type: application/json

2. Request Structure

{
  "strProviderCode": "NUFLIGHTS",
  "arrCarVouchers":    [ /* car voucher objects — see Section 4 */ ],
  "arrTicketVouchers": [ /* ticket voucher objects — see Section 4 */ ],
  "arrHotelVouchers":  [ /* hotel voucher objects — see Section 4 */ ],
  "arrOtherVouchers":  [ /* other voucher objects — see Section 4 */ ],
  "strCustomerCode":     "00001",   /* see Section 5 */
  "strCustomerCurrency": "INR",     /* see Section 5 */
  "strBranchCode":       "MAIN",    /* see Section 5 */
  "objPaymentInfo": { /* shared payment info for the single generated invoice — see Section 6 */ }
}
The endpoint does not accept an objCommonData block. Customer, branch, currency, ROE, precision, and created-by user are all resolved server-side from the three code fields plus the JWT. This mirrors the way the per-type third-party endpoints work.
All four voucher arrays are optional individually, but at least one must contain items. Sending an empty payload (or all four arrays empty) returns NO_RECORD_TO_SYNC.

3. Top-Level Fields

FieldTypeRequiredDescription

arrCarVouchers array Conditional Zero or more car voucher objects. Each element uses the exact same shape as one entry of arrVouchers in the Save Third-Party Car Voucher API.
arrTicketVouchers array Conditional Zero or more ticket objects. Each element uses the exact same shape as one entry of arrTickets in the Save Third-Party Tickets API.
arrHotelVouchers array Conditional Zero or more hotel voucher objects. Each element uses the exact same shape as one entry of arrVouchers in the Save Third-Party Hotel Voucher API.
arrOtherVouchers array Conditional Zero or more other-service voucher objects. Each element uses the exact same shape as one entry of arrVouchers in the Save Third-Party Other Voucher API.
strCustomerCode string Required Customer account code (e.g. "00001"). Resolved server-side to the full customer account row. See Section 5.
strCustomerCurrency string Required ISO currency code for the invoice (e.g. "INR", "USD"). Drives ROE and precision lookups.
strBranchCode string Required Branch code (e.g. "MAIN"). Resolved server-side to the full branch row.
objPaymentInfo object Required Shared payment information for the single consolidated invoice. See Section 6.
Conditional means: individually optional, but at least one of the four voucher arrays must be non-empty for the request to succeed.

4. Per-Type Voucher Object Fields

To avoid duplicating thousands of field definitions, the per-voucher object structures are documented in the existing per-type integration guides. This endpoint accepts exactly the same object shape for each voucher; only the enclosing array name differs.

Do not include a per-voucher objPaymentInfo on individual items — payment info is set once at the top level of this request and applies to the whole batch.

5. Customer, Currency & Branch

Instead of accepting a nested objCommonData block, the endpoint takes three code fields at the top level of the request and resolves the full context server-side. This keeps the payload aligned with the per-type third-party endpoints (which also work purely from codes).

What the caller sends

FieldTypeDescription
strCustomerCode string Account code registered against a customer partner. Rejected with INVALID_CUSTOMER_CODE if no matching active row exists.
strCustomerCurrency string ISO currency code — must exist in tbl_currency with a precision configured. Rejected with INVALID_CURRENCY otherwise.
strBranchCode string Branch code — must resolve to an active row in tbl_branch. Rejected with INVALID_BRANCH_CODE otherwise.

What the server derives

Resolved fieldSource
objCustomerAccount Looked up by strCustomerCode; carries intPk, strDefaultCurrency, strName, GDS-remark hints, and other account metadata.
objBranch Looked up by strBranchCode; carries intPk, strCode, strName.
dblRoe Fetched via the ROE settings usecase for strCustomerCurrency effective on objPaymentInfo.strDocumentDate. Falls back to 1 when the ROE settings return no rate (typically when the currency equals the base currency).
intPrecision Fetched via getAllCurrenciesPrecisionsValueDb, keyed by strCustomerCurrency.
strCreatedUser Resolved from source.intUserID (JWT) by joining tbl_user. Rejected with INVALID_USER if no active user row matches.
timCreated Request receipt timestamp from source.timReceived.
objDimensions null — this endpoint does not accept dimension overrides in the current iteration.
objSupplierAccount null — per-voucher supplier context is carried by each voucher payload and not required on the invoice level.

6. Payment Info (objPaymentInfo)

Payment context for the single generated invoice. Because this endpoint always generates one consolidated invoice, objPaymentInfo is required (unlike the per-type endpoints, where it is optional).

FieldTypeRequiredDescription
strDocumentDate string (date) Required Invoice document date.
strModeOfSale string Required Payment mode. Accepted values: "Credit", "Cash", etc. — matches the tenant's payment configuration.
arrPaymentOptions array Required At least one payment option describing how the invoice is settled. Same shape as in the per-type docs. The sum of dblTranCurDebitedAmount across all options is treated as the invoice total in the current iteration (see note under Section 7).

7. Processing Flow

  1. Open a single database transaction.
  2. For each non-empty voucher array, invoke the corresponding per-type third-party save flow. Each per-type flow runs its normal validations and credit-limit checks. If any voucher trips a credit-limit block, FAILED_DUE_TO_CREDIT_LIMIT_EXCEEDED is thrown, the transaction is rolled back, and no invoice is generated.
  3. Collect all successfully saved voucher numbers into a single list, tagged with their service type (C, T, H, O).
  4. Call the invoice generator once with the combined list, objCommonData, and objPaymentInfo.
  5. Commit the transaction and return the single strDocumentNumber together with each per-type save response for reference.
This endpoint is all-or-nothing across the whole batch: if any voucher in any type fails validation, or if invoice generation fails, the whole transaction rolls back and no voucher persists. Unlike the per-type third-party endpoints, this endpoint does not support partial success — mixed-batch invoices must not straddle partial failure.
Payment total check is currently skipped. Because real per-voucher totals are only computable after the per-type flows finish, this iteration stubs dblTotalClientAmount with the sum of arrPaymentOptions[].dblTranCurDebitedAmount, which makes the total-vs-debit equality check in validatePaymentInfo pass trivially. Payment-format validations (dates, batch numbers, mode of sale) still run as normal. A future iteration will compute totals from the saved vouchers and re-enable the equality check.

9. Responses

Success (HTTP 200)

{
  "strDocumentNumber": "INV/2026/07/000123",
  "objCarResponse":    { "Success": { "Sale": ["CAR-000045"], "Refund": [] } },
  "objTicketResponse": { "Success": { "Sale": ["176-2345678901"], "Refund": [] } },
  "objHotelResponse":  { "Success": { "Sale": ["HTL-000123"], "Refund": [] } }
}

strDocumentNumber is the single consolidated invoice covering every successfully saved voucher across all four types. The per-type response objects (objCarResponse, objTicketResponse, objHotelResponse, objOtherResponse) mirror the shape returned by the corresponding per-type endpoint and are present only for the types included in the request. Success is only returned when every voucher across every type passes validation — see below for the failure case.

Any-voucher Failure (rolled back)

If any voucher in any of the four types fails validation, the endpoint throws NO_RECORD_TO_SYNC, the transaction is rolled back, and no voucher persists. objDetails.objFailed carries the per-service-type failure roll-up so the caller can identify which vouchers were bad.

{
  "errCommon": [
    {
      "strMessage": "NO_RECORD_TO_SYNC",
      "objDetails": {
        "objFailed": {
          "T": { "Sale": { "176-9999000001": ["IMPORTING_ERRORS.INVALID_TRANSACTIONAL_REMARK_MASTER_Trip Purpose"] } }
        }
      }
    }
  ]
}

objFailed keys are the service-type letters C (car), T (ticket), H (hotel), O (other). Each service-type entry contains optional Sale and Refund maps of { voucherNumber: [reasons] }. The successful vouchers of other types (if any) are also rolled back — nothing is persisted.

If the payload contains no vouchers at all (every array missing or empty), the same NO_RECORD_TO_SYNC is returned but with an empty objDetails.
